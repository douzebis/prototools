# 0146 — reproto: emit a minimal `buf.yaml` alongside decompiled `.proto` output

Status: implemented
Implemented in: 2026-07-18
Refs: docs/specs/0145-protolens-neovim-lsp-config.md (bundled Neovim config
      that starts `buf lsp serve` with an LSP `root_dir`), 2026-07-18
      investigation session (live-verified against real `buf` 1.59.0 —
      `buf lsp serve` ignores the LSP `rootUri`/`root_dir` entirely for
      import resolution; without a `buf.yaml` at the true import root, it
      falls back to resolving imports relative to the *opened file's own
      directory*, which fails for any import reaching outside that
      directory — reproduced with a real headless Neovim + `buf lsp
      serve` round-trip; adding a minimal `buf.yaml` at the root fixed it)
App: reproto

## Background

Spec 0145 bundles a Neovim config that starts `buf lsp serve` and passes it
an LSP `root_dir`, on the assumption that `buf` would honor it for
resolving `.proto` imports across files. Live investigation (headless
Neovim + real `buf`) showed this assumption is false: `buf lsp serve`
never looks at `rootUri`/`workspaceFolders` for import resolution at all
— confirmed by getting byte-for-byte identical diagnostics whether
`root_dir` was set correctly or left `nil`. The only thing that actually
fixes cross-file import resolution is a `buf.yaml` discoverable by walking
up from the opened file — confirmed by adding a minimal one and seeing
diagnostics drop from 2 errors to 0 on an otherwise-unchanged corpus.

`reproto -O <dir>` is the primary way `.proto` corpora consumed by
protolens's `v` Neovim handoff get produced in the first place. Since
`reproto` already fully controls and owns the output directory it writes
into, it is the natural, low-friction place to drop a `buf.yaml` next to
the `.proto` files it generates — no action required from the user, and
no risk of clobbering a hand-written `buf.yaml` in a directory reproto
didn't create from scratch.

## Goals

- **G1 (write a minimal `buf.yaml` at the `-O` root).** After
  `_phase7_output` finishes writing `.proto` files (or would have, absent
  `--dry-run`) into `out_repo`, write:
  ```yaml
  # Auto-generated by reproto — lets `buf lsp serve` (used by protolens's
  # Neovim integration, spec 0145) resolve imports rooted at this
  # directory. Safe to edit or delete.
  version: v2
  modules:
    - path: .
  ```
  to `out_repo / "buf.yaml"`.

- **G2 (respect `--dry-run`).** Skipped whenever `ctx.dry_run` is true —
  mirrors the existing `.proto`/`.pb` write guards in `_phase7_output`
  (`--dry-run` means "do not actually create ... files", full stop; this
  is consistent with that existing contract, not a new one).

- **G3 (never overwrite an existing `buf.yaml`).** Only written when
  `out_repo / "buf.yaml"` does not already exist — `-O <dir>` may point at
  a directory the user re-runs `reproto` into repeatedly, or one where
  they've hand-customized `buf.yaml` (extra lint rules, a second module,
  etc.); reproto must never silently clobber that.

- **G4 (schema-db-only mode unaffected).** `reproto()` (in `reproto.py`)
  calls `_phase7_output(ctx, out_repo or Path('.'))` even when `-O` was
  not given, forcing `ctx.dry_run = True` for that call so no `.proto`
  files leak into the process's cwd (`Path('.')` placeholder). G2's guard
  already covers this — no separate condition needed.

## Non-goals

- N1: No CLI flag to opt out of `buf.yaml` generation — it is inert
  (empty/no-op) for any consumer other than `buf`, and free to delete
  (documented in the file's own header comment).
- N2: No attempt to merge/update an already-existing `buf.yaml`'s
  contents (e.g. adding a `lint`/`breaking` section) — G3 leaves any
  pre-existing file completely untouched.
- N3: No change to protolens/Neovim (spec 0145) — this is a reproto-only
  change; protolens's `.proto` corpora simply gain a `buf.yaml` the next
  time they're regenerated via `reproto -O`.
- N4: No validation that the generated `buf.yaml` is itself lint-clean
  under `buf`'s own rules (e.g. `PACKAGE_DIRECTORY_MATCH`) — those are
  properties of the decompiled `.proto` content, unrelated to whether
  `buf.yaml` exists at all, and out of scope here.

## Specification

### `reproto/src/reproto/phases.py`

At the end of `_phase7_output`, after the `for re_fdp in summoned:` loop:

```python
    if not ctx.dry_run:
        buf_yaml_path = out_repo / "buf.yaml"
        if not buf_yaml_path.exists():
            buf_yaml_path.write_text(
                "# Auto-generated by reproto — lets `buf lsp serve` (used by\n"
                "# protolens's Neovim integration, spec 0145) resolve imports\n"
                "# rooted at this directory. Safe to edit or delete.\n"
                "version: v2\n"
                "modules:\n"
                "  - path: .\n"
            )
```

## Test plan

- New test asserting `buf.yaml` is written at `out_repo`'s root after a
  normal `reproto -O <dir>` run, with the expected minimal content.
- New test asserting `buf.yaml` is *not* written when `--dry-run` is
  passed.
- New test asserting a pre-existing `out_repo/buf.yaml` (arbitrary
  content) is left untouched by a subsequent `reproto -O <dir>` run.
- Manual: re-run spec 0145's own QA checklist step 6 ("no
  `buf.yaml`/`.git` ancestor" fallback case) against a freshly
  `reproto -O`'d corpus — `gd` should now resolve cross-file imports
  without needing the `PROTOTEXT_PROTO_ROOT` fallback at all, since
  `buf.yaml` is now always present.
